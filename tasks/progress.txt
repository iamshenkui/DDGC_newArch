## Codebase Patterns
- Monster families use `MonsterFamilyRegistry` with `FamilyId` keys — each migration slice calls `registry.register(family)` once
- DDGC has exactly two tiers: `MonsterTier::Common` and `MonsterTier::Boss` — no elite tier exists in the data
- The `monsters` module is declared in `lib.rs` alongside `content`, `encounters`, `heroes`, `parity`, `run`, `scenarios`, `trace`
- `build_registry()` in `src/monsters/mod.rs` registers all migrated MonsterFamily entries — each slice adds one `registry.register()` call
- Summon bosses (gambler, necrodrake, rotvine_wraith) use `FamilyRole::Summoner` and `apply_status("summon_XXX", Some(N))` — actual spawn behavior is a game-gap
- `EffectNode` does not have a `status_magnitude` field — available fields are `kind`, `params`, `conditions`, `status_kind`, `status_duration`, `attribute_key`
- Boss encounter packs for summon-type bosses start with only the main boss unit — minions are not pre-placed in the pack
- `register_content()` in `src/content/monsters/mod.rs` wires each family's archetype + skills into ContentPack — each slice adds `pub mod` + registration calls
- DDGC `.def` percentages map to `Archetype.defense` as decimals (e.g., 7.5% → 0.075)
- DDGC damage ranges are averaged to a single float (e.g., 20–28 → 24.0)
- Position-based targeting (`.launch`, `.target`) is not modeled — use `AllEnemies`, `AllAllies`, or `SelfOnly` as game-gap approximations
- DDGC `@1234` target prefix (any ally rank, not self) is approximated as `TargetSelector::AllAllies` (includes self — game-gap)
- Movement skills (`.move N M`) are approximated as `EffectNode::push(N)` with `TargetSelector::SelfOnly`
- Monster archetypes use `CombatSide::Enemy`; heroes use `CombatSide::Ally`
- Module filenames must be snake_case — use `moth_mimicry_a.rs` not `moth_mimicry_A.rs`; but keep `FamilyId::new("moth_mimicry_A")` to match DDGC naming
- On WSL/case-insensitive filesystems, rename case-only via two-step `mv` (old → tmp → new)
- Summon/split skills are modeled as `apply_status("split", Some(N))` with `TargetSelector::SelfOnly` and cooldown matching DDGC AI brain — actual summon behavior is a game-gap
- Boss families use `MonsterTier::Boss`, `FamilyRole::Summoner`, and may consist of multiple units (main body + boss parts) — each part gets its own content file and registry entry
- Self-buff side effects embedded in offensive skills (DDGC `performer`-targeted effects) are dropped as game-gaps — framework applies all effects to the skill's target_selector result
- Encounter packs use `EncounterPackRegistry` with `PackId` keys — `build_packs_registry()` in `src/encounters/mod.rs` populates from per-dungeon pack functions
- Pack naming convention: `{dungeon}_{hall|room}_{NN}` for common packs (e.g., `qinglong_hall_01`), `{dungeon}_boss_{boss_name}` for boss packs (e.g., `qinglong_boss_azure_dragon`)
- Boss encounter packs use `PackType::Boss` — boss pack builder functions follow naming convention `{dungeon}_boss_packs()`
- `Dungeon` and `FamilyId` are re-exported via `pub use` in `encounters/packs.rs` so they're accessible through `pub use packs::*`
- HashMap iteration order is not deterministic — always sort `by_dungeon_and_type()` results by pack ID before index-based selection
- Legacy tutorial heroes (Crusader/Vestal) have ~30 HP vs DDGC-scale heroes (135-192 HP) — use DDGC-scale heroes for encounter battles
- Hero skill names can collide in ContentPack (e.g., "stun_skill" registered by both Hunter and Tank) — use unique skill names for reliable lookup
- Skills with empty effects vec fail `SkillDefinition::validate()` — even placeholder skills need at least one `apply_status` marker
- Multi-phase bosses (white_tiger) have clone units (A/B) with `FamilyRole::Support`/`Controller` and a final form (C) with `FamilyRole::Summoner`
- Corpse-type terrain units (white_tiger_terrain) use `MonsterType::Corpse`, SPD 0, HP 15, and a single placeholder skill
- Composite dual-body bosses (black_tortoise) register each body as a separate `MonsterFamily` — A body (Tank) and B body (Controller) with `FamilyRole::Tank` and `FamilyRole::Controller`
- When both bodies share DDGC skill names (call_roll, rain_spray), prefix with body identifier (tortoise_call_roll, snake_call_roll) to avoid ContentPack collisions
- Cross-body damage sharing (near_mountain_river/armor) uses `TargetSelector::AllAllies` + `apply_status("share_damage", Some(N))` — the actual damage redistribution is a game-gap
- The `$1234` conditional targeting notation (targets heroes with specific marks) is approximated as `AllEnemies` — same game-gap pattern as `#1234` AoE
- When a boss and its summon share a DDGC skill name (e.g., both skeletal_tiller and vegetable have `crop_rot_claw`), prefix with unit identifier (`tiller_crop_rot_claw`, `vegetable_crop_rot_claw`) to avoid ContentPack collisions
- AI brain skills not in the active rotation (e.g., skeletal_tiller's `crop_rot_claw`) are still registered in the skill pack — the AI priority system is not modeled in the framework
- Boss skills with `Debuff Frozen-30_2` effects are modeled as `apply_status("frozen", Some(N))` — the specific resist reduction value is a game-gap
- Cauldron-type captor units (egg_membrane) use `MonsterType::Cauldron` with placeholder skills — each variant (empty/full) is a separate `MonsterFamily` entry; the `captor_empty`↔`captor_full` state transition is a game-gap
- DDGC `.target` empty field on skills (e.g., ecdysis_metamorphosis) indicates `SelfOnly` targeting — model as `TargetSelector::SelfOnly`
- DDGC "riposte" skills are counter-attack triggers — prefix with unit identifier (e.g., `clam_riposte`) to avoid ContentPack collisions
- DDGC dual-effect skills (e.g., "fc Riposte" = riposte + prot buff) should be modeled as separate `apply_status` nodes in the same skill
- DDGC "Frozen" DoT effects are modeled as `apply_status("frozen", Some(N))` where N is duration — per-tick damage is a game-gap
- DDGC Stress Range values (e.g., "Stress Range 7-10") are averaged to a single value
- Cross-dungeon bosses (bloodthirsty_assassin, glutton_pawnshop) use `Dungeon::Cross` and `cross_boss_packs()` — they appear outside the four main dungeon regions
- Paired bosses (bloodthirsty_assassin + shadow) register both units as separate `MonsterFamily` entries with `FamilyRole::Bruiser` and `FamilyRole::Support` — HP-averaging mechanics are modeled as `apply_status("average_hp", None)` with `TargetSelector::AllAllies`
- Controller-mechanic bosses (glutton_pawnshop) use `FamilyRole::Controller` and apply tag-based debuffs via `apply_status("gp_control"/"gp_tag_bleed"/"gp_tag_blight"/"gp_tag_debuff", Some(N))` — the conditional tag-trigger behavior is a game-gap
- The `$1234` conditional targeting notation (targets heroes with specific marks/tags) is approximated as `AllEnemies` — same game-gap as `#1234` AoE and `~1234` AoE
- Target resolution from `TargetSelector::AllEnemies`/`AllAllies` returns targets in HashMap-iteration order — must sort by ActorId for deterministic trace output
- `EncounterResolver` is the primary entry point for gameplay battles — both `main.rs` and `flow.rs` use `resolve_pack()` for combat and `resolve_boss_pack()` for boss rooms
- The `first_battle` scenario module and Bone Soldier/Necromancer archetypes are legacy content — kept for parity tests and transitional fallback, but no longer used in any gameplay entry point
- Fallback battle paths use `pack_id: "fallback_transitional"` to distinguish from real encounter battles in traces
