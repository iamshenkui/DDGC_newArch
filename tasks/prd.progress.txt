## Codebase Patterns
- SEMANTIC_PARITY.md and SEMANTIC_GAPS.md are the parity vocabulary docs; all parity stories reference them
- MIGRATION_BLOCKERS.md is the authoritative source for blocker classification; SEMANTIC_GAPS.md cross-references it
- game_ddgc_headless uses `cargo check` for typecheck and `cargo test` for tests
- DDGC hero stats come from equipment .bytes files, not base class definitions — heroes have zero base stats; level 0 weapon/armor apply all combat attributes
- DDGC .bytes files are actually plaintext (despite extension) — use `strings` or direct reads
- Skill damage values are calculated as: base_weapon_avg × damage_modifier_percent (e.g., -50% = half, -100% = zero, 0% = full)
- All skills use TargetSelector::AllEnemies or AllAllies — single-target selection is a known game-gap (B-006)
- Utility-only skills (self-move, self-buff, oracle) approximated as heal(1.0) placeholder — framework has no movement or buff EffectNodes
- ContentPack registers all hero archetypes and skill packs — new hero content goes in src/content/heroes/
- BaseHeroVariant in src/heroes/base.rs connects family class_id to content via match dispatch
- White variant skill IDs use "w_{base_id}_{skill_name}" pattern — ensures ContentPack uniqueness and avoids collision with base skills
- Black variant skill IDs use "b_{base_id}_{skill_name}" pattern — same convention as white, ensures uniqueness across all variant types
- DDGC white (+1) and black (+2) variants have identical base stats to their base counterparts — differences are exclusively in skill effect chains
- EffectNode does not implement PartialEq — use effect count comparison to verify skill differences in tests
- Conditional/procedural effects (divine count, mark refresh, % chance secondary DoT) approximated as guaranteed effects within framework constraints
- FamilySkillResolver in src/heroes/skills.rs dispatches (base_id, chaos_mode) to variant content modules — same match pattern as BaseHeroVariant/WhiteHeroVariant/BlackHeroVariant
- FamilyStatusRegistry derives variant status profiles from skill definitions at runtime — not hardcoded, stays in sync with content changes
- Diviner base variant has zero ApplyStatus effects (all damage/heal) — white/black variants add tagged and stun statuses
- DDGC monsters have no "elite" tier — only Common (size 1, 1 turn/rd) and Boss (size 2+, 2-3 turns/rd, tag="boss")
- Common monsters use difficulty tiers (_1/_2/_3) scaling HP ~1.4×/1.8× and SPD +1/+2 per tier
- DDGC monster data files are .txt DSL format under Assets/Resources/Data/Monsters/ — not JSON/YAML
- Boss encounters use 7 structural patterns: shared_health, multi_body, multi_phase, captor, summon, controller, paired
- Monster .txt files specify enemy_type (beast/eldritch/unholy/man/cauldron/corpse), optional tag/shared_health/riposte_skill/controller fields
